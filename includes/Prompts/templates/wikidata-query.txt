# OUTPUT FORMAT

# SPARQL GRAMMAR REFERENCE

## Query shape
SELECT [DISTINCT] <var-list> WHERE { <GroupGraphPattern> } [solution-modifiers]

## Triple patterns
- Basic: `subject predicate object .`
- Predicate list (same subject): `?s p1 ?o1 ; p2 ?o2 .`
- Object list (same subject + predicate): `?s p ?o1 , ?o2 .`

## Property paths
- `/` sequence:        `wdt:P31/wdt:P279*`
- `^` inverse:         `^wdt:P40` (the inverse direction)
- `*` zero-or-more, `+` one-or-more, `?` zero-or-one
- `|` alternative:     `wdt:P50|wdt:P170`
- `!` negation:        `!wdt:P31`

## Group patterns inside WHERE
- `OPTIONAL { ... }`              — bind if present, else leave unbound
- `{ A } UNION { B }`             — disjunction
- `MINUS { ... }`                 — subtract matches
- `FILTER (expr)`                 — constrain bindings already produced in the enclosing group
- `FILTER EXISTS { ... }` / `FILTER NOT EXISTS { ... }`
- `BIND(expr AS ?var)`            — compute a value
- `VALUES ?v { wd:Q1 wd:Q2 }`     — inline value list
- Subquery: `{ SELECT ... WHERE { ... } LIMIT n }`

## Built-in functions usable in FILTER / BIND
STR, LANG, DATATYPE, BOUND, IF, COALESCE, REGEX, CONTAINS, STRSTARTS, STRENDS,
YEAR, MONTH, DAY, HOURS, NOW, xsd:dateTime("..."), xsd:integer, xsd:decimal.
Operators: + - * /  =  !=  <  <=  >  >=  &&  ||  !

## Aggregates (require GROUP BY for every non-aggregated select var)
COUNT(*), COUNT(DISTINCT ?x), SUM(?x), AVG(?x), MIN(?x), MAX(?x), SAMPLE(?x),
GROUP_CONCAT(?x; SEPARATOR=", "). Filter aggregates with HAVING(expr).

## Solution modifiers (must appear in this order, after the WHERE block)
GROUP BY ... → HAVING(...) → ORDER BY ASC/DESC(...) → OFFSET n → LIMIT n

## Wikidata prefixes (already declared — never re-declare)
- wd:        entity                  (wd:Q937)
- wdt:       truthy direct claim     (wdt:P569)
- p:         statement node          (p:P39 → ?stmt)
- ps:        statement value         (ps:P39 wd:Q11696)
- pq:        statement qualifier     (pq:P580 ?startDate)
- psv:/pqv:  value-node (precision, units, calendar)
- wikibase:, bd:, rdfs:, schema:, skos:, xsd:

## Common pitfalls — avoid these
- `wdt:` cannot reach qualifiers. Use `p:/ps:/pq:` whenever you need start date, end date, rank, point in time, etc.
- Every non-aggregated SELECT variable must appear in GROUP BY when any aggregate is used.
- FILTER only constrains bindings already produced in the same group; put it after the patterns it references.
- OPTIONAL leaves variables unbound on miss — guard downstream with BOUND(?x) or COALESCE.
- Date comparisons need typed literals: `FILTER(?d > "1900-01-01T00:00:00Z"^^xsd:dateTime)` and `FILTER(YEAR(?d) < 1900)`.
- Quantity values from `wdt:` are bare numbers; for units use `psv:` with `wikibase:quantityAmount` and `wikibase:quantityUnit`.
- Property paths cannot be used inside SERVICE wikibase:label; resolve labels via separate variables.
- Aggregating label variables (e.g., `GROUP_CONCAT(?valueLabel)`) causes them to return as empty strings. To aggregate labels, you MUST resolve them inside a subquery BEFORE applying `GROUP BY` and `GROUP_CONCAT` in the outer query.

# QUERY RULES (apply on top of the grammar)
1. SERVICE wikibase:label MUST be the LAST clause INSIDE the WHERE block, immediately before the closing `}`. Never place it outside WHERE. Template:
   SELECT ... WHERE {
     ... triple patterns ...
     SERVICE wikibase:label { bd:serviceParam wikibase:language "[AUTO_LANGUAGE],{{lang}},en" }
   }
2. For every entity you want a label for, project a `?xLabel` variable (?itemLabel, ?valueLabel, ?countryLabel, ...). The label service binds these automatically.
3. Always include LIMIT (default 10, max 50) unless using LIMIT 1 for a superlative.
4. SELECT only. Never use CONSTRUCT / ASK / DESCRIBE.
5. Item placeholders → `wd:PLACEHOLDER`; property placeholders → `wdt:PLACEHOLDER` (or `p:`/`ps:`/`pq:` with the SAME placeholder when qualifiers are needed).
6. Open-ended "tell me about X / who is X / describe X / what do you know about X" — decide which sub-case applies:

   (6a) The question names specific attributes ("what's Apple's revenue and CEO?", "where and when was Marie Curie born?") → resolve each named attribute as a property entity and query those PIDs directly. Do NOT use the grouped-dump form below.

   (6b) The question is truly open-ended with no named attributes ("tell me about Apple", "who is Marie Curie?") → fetch all direct claims for the entity, GROUPED BY property so high-cardinality properties (e.g. Apple Inc.'s hundreds of MAC address OUI blocks) don't crowd out everything else. Do NOT query skos:altLabel or rdfs:label (those return only names). Use:
   SELECT ?propLabel (GROUP_CONCAT(DISTINCT ?valLabel; SEPARATOR=", ") AS ?values) WHERE {
     {
       SELECT ?propLabel ?valLabel WHERE {
         wd:PLACEHOLDER ?p ?value .
         ?prop wikibase:directClaim ?p .
         SERVICE wikibase:label { bd:serviceParam wikibase:language "[AUTO_LANGUAGE],{{lang}},en" . ?prop rdfs:label ?propLabel . ?value rdfs:label ?valLabel }
       }
     }
     UNION
     {
       wd:PLACEHOLDER schema:description ?valLabel .
       FILTER(LANG(?valLabel) = "en")
       BIND("description" AS ?propLabel)
     }
   } GROUP BY ?propLabel LIMIT 50
   } GROUP BY ?propLabel LIMIT 50
7. Conditional questions ("X that are Y", "X born in Y", "X with more than N", "X that also Z") → chain triple patterns for AND, use FILTER for comparisons, OPTIONAL for nullable data. If you cannot resolve every entity in one shot, set "status":"NEEDS_MORE".
8. Ordinal / superlative ("first", "last", "earliest", "most recent", "oldest", "youngest", "tallest", "highest", "lowest", "most X") → ORDER BY the relevant date or numeric var, LIMIT 1 (unless a ranked list is requested):
   - ASC for first / oldest / earliest / lowest / smallest
   - DESC for last / most recent / latest / highest / largest
9. Qualifier-bearing statements (start date, end date, rank, point in time on "position held", "award received", "member of", etc.) → use the full statement form:
     ?item p:PROP ?stmt .
     ?stmt ps:PROP wd:VALUE ;
           pq:QUALIFIER ?qval .
   The same PID placeholder works for `wdt:`, `p:`, `ps:`, `pq:`.
10. Class-based searches ("all X", "every X", "any X" — especially broad categories like statue, mammal, programming language, mountain) → traverse subclasses with `wdt:P31/wdt:P279*`, not `wdt:P31` alone.
